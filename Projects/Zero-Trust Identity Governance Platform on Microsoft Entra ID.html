Module 1: Application access — zero-secret authentication

Two ways to eliminate stored credentials, chosen by where the caller actually runs.

Zero-secret CI/CD authentication (Workload Identity Federation)

sequenceDiagram
    participant GA as GitHub Actions
    participant GH as GitHub OIDC Provider
    participant Entra as Entra ID
    participant Azure as Azure Resource

    GA->>GH: Request OIDC token for this workflow run
    GH-->>GA: Short-lived signed JWT
    GA->>Entra: Present JWT (client-id, tenant-id — no secret)
    Entra->>Entra: Validate JWT against Federated Credential trust
    Entra-->>GA: Issue Azure AD access token
    GA->>Azure: Call ARM API with access token
    Note over GA,Azure: Token is scoped to this exact workflow run and repo/branch — cannot be replayed elsewhere

Module 2: Conditional Access — layered by risk, not one-size-fits-all

Six policies, each matched to a specific risk scenario. Verification strength scales with the sensitivity of what's being protected.

PolicyScopeControl
CA001a — Baseline MFAAll users (break-glass excluded)Require MFA (any method)
CA001b — Privileged role hardeningPrivileged RolesRequire phishing-resistant MFA (FIDO2 / Passkey / certificate)
CA002a — Sign-in risk responseAll usersMedium/high sign-in risk → Require MFA
CA002b — User risk responseAll usersHigh user risk → Require password change (SSPR)
CA003 — Legacy auth blockingAll usersBlock legacy authentication and browser clients
CA004 — Unmanaged device restrictionAll usersSession controls — no download

Why two different MFA strengths, not one

A compromised standard user account exposes that user's own scope. A compromised privileged account exposes the tenant. The two shouldn't require the same proof of identity.

Module3: Privileged access management (PIM)

Every activation is time-bound, justified, and auditable — nothing is standing.

What this project demonstrates

  • 1. Application access management Zero-secret Workload Identity Federation between GitHub Actions and Entra ID via OIDC, replacing long-lived client secrets with per-run federated tokens scoped to a specific repository, branch, and environment.
  • 2. Adaptive authentication & access Six layered Conditional Access policies matched to specific risk scenarios: baseline MFA for all users, phishing-resistant authentication strength for privileged roles, sign-in and user risk response tied to Identity Protection, legacy authentication blocking, and session controls for unmanaged devices — with a protected break-glass account excluded from every policy.
  • 3. Privileged access governance Conversion of high-privilege roles from Permanent Active to PIM Eligible, spanning both Microsoft Entra directory roles (Global Administrator, Privileged Role Administrator...) and Azure resource roles (Key Vault Secrets Officer, Automation Contributor...) — with enforced MFA, activation duration limits, and a documented fix for a resource-group-level role inheriting further than intended into child resources.
  • 4. Identity lifecycle governance Two Access Package lifecycle models tested end to end (hard 1-hour expiry vs. standing access with recurring Access Review), plus full Joiner-Mover-Leaver Lifecycle Workflow automation — including the department-change Mover workflow that most implementations skip, which is where real-world privilege creep actually accumulates.

Questions & Answers

  • Why Workload Identity Federation over Managed Identity? Tested both against the same Key Vault: Managed Identity requires the caller to be an Azure-hosted resource, since Azure binds the identity at the infrastructure layer — GitHub Actions runs on GitHub's own servers, so it doesn't qualify. Federation solves this by establishing trust at the OIDC protocol layer instead of resource ownership, making it the only viable zero-secret option for external CI/CD platforms.
  • Why risk signals live inside Conditional Access rather than legacy Identity Protection policies? Microsoft's current guidance consolidates risk-based conditions into Conditional Access for a single, unified policy surface rather than two separate enforcement points.
  • Why Eligible for every sensitive role? Standing privilege is the single largest reduction in blast radius available without impacting legitimate work — the cost is a few seconds of activation friction per use.
  • Why extending PIM beyond directory roles to Azure resource roles? Most PIM implementations only cover Microsoft Entra directory roles (Global Administrator, etc.). This project also brought Azure resource-level roles into PIM — Key Vault and Automation Account access — since over-permissioned resource access is at least as common a risk in practice.
  • Why RBAC to resource scopes? After assigning Contributor at the resource group level, the same role appeared — inherited, not duplicated — on every child resource inside it, including the Automation Account where a narrower role was intended. Removed the resource-group-level assignment and kept only resource-scoped roles, since RBAC assignments at a parent scope silently propagate to everything underneath.
  • Why the break-glass account still needs MFA(passkey)? Originally configured with no MFA at all, per older guidance. Microsoft's platform-level mandatory MFA enforcement for the Azure portal, Entra admin center, and Intune admin center (phased in from October 2024) applies even to emergency access accounts and cannot be bypassed via Conditional Access exclusions — it's enforced at the client application layer, independent of tenant policy. Resolved by registering a FIDO2 passkey on the break-glass account, per Microsoft's current recommendation for emergency accounts.
  • Why Global Administrator wasn't enough to configure Lifecycle Workflows? Attempting to open Lifecycle Workflows while signed in as Global Administrator returned a 401 error. The feature requires the dedicated Lifecycle Workflows Administrator role rather than trusting GA's implied permissions — a small but real example of Microsoft's own newer governance features moving toward least-privilege role design rather than a catch-all admin role.
  • Why the default Microsoft Leaver template got a risky task must be removed before use? The stock Leaver template included a permanent account-deletion task. Testing it with Days from event: 0 deleted the test account immediately rather than just disabling it — recovered via Entra ID's 30-day soft-delete restore. Disablement and token revocation should happen immediately on a leave date; permanent deletion is a separate, later-stage decision that shouldn't be bundled into day-zero offboarding by default.
